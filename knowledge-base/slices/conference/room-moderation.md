# 会议会控与秩序管理

## 功能说明

会议会控与秩序管理负责会议进行中的动态秩序控制，覆盖全体禁用设备、全体禁言、单独禁言、关闭指定成员设备，以及设备申请和设备邀请这类会中协作。它解决的是“谁能在会中限制哪些交互、受限成员如何恢复设备能力”这些问题，关注的是运行时规则，不是创建前默认配置，也不是成员角色本身。

## 核心概念

### 角色与操作

| 角色 | 关键操作 | 说明 |
|------|----------|------|
| 房主 | 发起全员规则、成员级限制和设备协作 | 可调用 `disableAllDevices`、`disableAllMessages`、关闭成员设备，并处理设备申请或邀请成员开启设备 |
| 管理员 | 执行会中规则和成员级会控 | 也可调用 `disableAllDevices`、`disableAllMessages`、关闭成员设备，并处理设备申请和设备邀请 |
| 普通成员 | 感知规则并响应协作 | 在设备或消息能力受限时遵守规则，也可申请开启设备或响应管理员邀请 |
| 聊天 / 设备 / 共享模块 | 消费会控结果 | 禁言、禁设备、禁共享等规则最终会作用到输入区、设备按钮和共享入口 |

### 事件流

| 阶段 | 参与方 | 关键动作 |
|------|--------|----------|
| 发起控制 | 房主 / 管理员 | 发起全员规则、单成员限制或设备协作动作 |
| 规则同步 | 房间状态 | 房间级规则、成员级限制和待处理协作状态同步到各端 |
| 成员响应 | 普通成员 | 根据当前限制调整输入和设备操作，必要时申请或响应设备邀请 |
| 结果回流 | 客户端 / 业务层 | 申请被批准、被拒绝、超时、已被他人处理等结果回流到界面 |
| 规则解除 | 房主 / 管理员 | 取消或调整当前会控状态，并收口相关提示和入口 |

### 状态与数据

| 数据 / 状态 | 说明 |
|-------------|------|
| 全体设备禁用状态 | 控制房间内普通成员是否还能自行开启麦克风、摄像头或共享能力；房主和管理员通常不受该限制 |
| 管理端房间级会控读模型 | 供房主 / 管理员按钮、菜单文案和二次确认提示回显当前全体静音 / 禁画 / 禁共享 / 禁聊状态；不应只存在于一次点击的临时变量中 |
| 全体消息禁用状态 | 控制普通成员聊天输入区是否可用；房主和管理员通常仍保留必要的管理与沟通能力 |
| 单成员消息禁用状态 | 控制某个成员是否被单独禁言 |
| 指定成员设备关闭状态 | 用于处理单成员摄像头、麦克风或共享被关闭的结果 |
| 待处理设备申请 | 表示当前仍待管理员处理的设备开启申请 |
| 待处理设备邀请 | 表示当前仍待成员响应的设备开启邀请 |
| 当前用户角色 | 决定是否拥有会控入口、审批能力和邀请能力 |

### 状态机

```text
default-room-state
  → moderation-updated
  → participant-restricted
  → request-or-invitation-pending
  → result-synced
  → moderation-released
  → default-room-state
```

## 最佳实践

### ✅ ALWAYS

1. **把创建前默认规则与会中动态会控分开建模** —— 默认值来自 `room-config`，运行时调整来自当前 slice。
2. **区分房间级规则、成员级限制和设备协作链路** —— 全体禁言、全体禁用设备属于房间级规则；单独禁言、关闭成员设备属于成员级限制；设备申请和邀请是受限后的协作恢复链路。
3. **让会控结果直接作用到真实交互路径** —— 禁言要禁用输入区，禁设备要限制真实开启动作，禁共享要限制共享入口，而不是只改文案。
4. **让管理端按钮体现当前房间级规则，并支持 toggle 解除** —— “全体静音 / 解除全体静音”、“全体禁言 / 解除全体禁言”都应围绕当前状态成对出现，而不是只提供单向动作。
5. **同时处理入口显隐、状态回流和结果反馈** —— 会控按钮、成员列表标识、待处理列表和提示信息应围绕同一份会控状态同步更新。
6. **让房间级会控状态可恢复** —— 页面刷新、重进房或管理面板重新挂载后，房主 / 管理员侧按钮应从房间状态同步结果或业务侧会控快照恢复，而不是默认回到未禁用态。
7. **明确 `disableAllDevices` 的作用范围** —— `disableAllDevices` 是房间级规则，可作用于麦克风、摄像头和屏幕分享；通常只约束普通成员，房主和管理员不受该限制。
8. **区分房间级禁用与成员级关闭的恢复路径** —— 房间级禁用期间，普通成员不应再直接开启对应能力，入口应表现为 disabled，并按产品需要走申请开启链路；若只是被单独关闭摄像头、麦克风或共享，在没有叠加房间级禁用时，成员得到明确提示后仍可再次主动开启。
9. **明确全体消息禁用的生效范围** —— `disableAllMessages` 通常只约束普通成员；房主和管理员仍需保留必要的会中管理与沟通能力。
10. **对设备申请和邀请做好超时、取消和重复处理收口** —— 多管理员协作时，申请或邀请可能已被他人处理，界面应及时关闭无效入口。

### ❌ NEVER

1. **不要把角色治理和房间会控混成一套逻辑** —— 设置管理员、转让房主属于成员治理，不属于会控本身。
2. **不要把房主 / 管理员的房间级会控按钮做成只有“禁用”没有“解除”的单向操作** —— 真实产品中的全员静音、禁画、禁聊都应支持回退。
3. **不要只在界面上禁用按钮而不限制真实能力** —— 规则必须落到聊天发送、设备开启和共享流程的真实执行路径。
4. **不要把全体禁用状态只存在当前页面内存里** —— 页面刷新、重进房或会控面板重新挂载后，仍应能恢复当前房间级规则。
5. **不要把设备打不开一律归因到浏览器或系统问题** —— 还要检查是否被全体禁用、被管理员关闭，或需要走设备申请 / 邀请链路。
6. **不要让普通成员暴露房间级会控入口** —— 会控能力应严格受房主 / 管理员权限边界约束。

## 排障指南

### 常见问题

| 问题 | 表现 | 处理建议 |
|------|------|----------|
| 会控面板生效但成员侧无变化 | 已开启全体禁言、禁设备或禁共享，但成员侧入口仍可用 | 检查聊天、设备、共享模块是否直接消费了会控状态 |
| 房间未全体禁言但某个成员不能发消息 | 只有个别成员输入区不可用 | 检查是否存在单独禁言状态，而不是误判为聊天模块异常 |
| 设备申请长期不收口 | 申请已取消、超时或被他人处理，但界面仍显示待处理 | 检查申请状态是否和超时、取消、已处理结果同步更新 |
| 管理入口展示错误 | 普通成员也能看到房间级会控入口 | 检查当前用户角色与会控入口显隐是否绑定正确 |
| 刷新后管理端按钮状态丢失 | 房间里其实仍处于全体静音 / 禁画 / 禁聊，但房主端按钮又显示成初始态 | 检查管理端按钮是否从房间状态同步结果或业务侧会控快照恢复，而不是只依赖页面内 `ref` |
| 规则与默认配置混淆 | 创建前配置的默认禁麦被误当成会中临时会控 | 检查 `room-config` 与 `room-moderation` 是否做了清晰分层 |

### 排障流程

```text
发现 会议会控与秩序管理 相关问题
├── 第 1 步：确认问题属于会中动态规则，而不是创建前默认配置或成员角色治理
├── 第 2 步：检查当前用户是否具备房主 / 管理员权限，并据此展示会控入口或审批入口
├── 第 3 步：确认禁聊、禁设备、禁共享、设备申请 / 邀请等状态是否已直接联动到聊天、设备和共享模块
└── 第 4 步：若仍异常，再回查 room-config / participant-management / room-chat / device-control / screen-share 的衔接是否正确
```
