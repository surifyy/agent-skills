# 参会人管理与角色治理

## 功能说明

参会人管理与角色治理负责会议中的成员级管理，解决“谁是房主、谁是管理员、谁可以管理别人、哪些成员可以继续留在会议里、成员名片和业务信息由谁维护”这类问题。它主要关注角色变化、成员治理动作和结果收口，不负责成员列表展示，也不等同于全体禁麦、禁画、禁共享、禁聊这类房间级会控。

## 核心概念

### 角色与操作

| 角色 | 关键操作 | 说明 |
|------|----------|------|
| 房主 | 设置 / 撤销管理员、转让房主、移出成员、处理成员设备相关协作 | 通用会议中同一时间通常只有一个房主，是成员治理的最高权限角色；从角色权限视角也可处理关闭成员设备、设备申请和设备邀请等协作 |
| 管理员 | 协助管理在会成员与设备协作 | 可执行部分成员治理动作，也可参与关闭成员设备、处理设备申请、邀请成员开启设备等协作，但不能设置管理员、撤销管理员或转让房主 |
| 普通成员 | 查看自身角色、维护自身信息、申请开启受限设备 | 通常不能治理他人，但可维护自己的名片，并在设备受限时发起开启申请或响应管理侧邀请 |

### 关键治理动作

| 动作 | 谁可以发起 | 结果 | 说明 |
|------|------------|------|------|
| `setAdmin` | 房主 | 把普通成员设为管理员 | 只有房主可以发起 |
| `revokeAdmin` | 房主 | 撤销管理员身份 | 只有房主可以发起 |
| `transferOwner` | 房主 | 把房主身份转交给其他成员 | 转让后新旧房主权限应立即切换 |
| `kickUser` | 房主 / 管理员 | 指定成员离开当前会议 | 结果应同步回成员列表和房间状态 |
| 设备相关协作 | 房主 / 管理员 / 普通成员 | 围绕设备限制进行成员间协作 | 包括关闭成员设备、处理设备申请、邀请成员开启设备，以及普通成员申请开启受限设备；详细规则见 `conference/room-moderation` |
| `updateParticipantNameCard` | 成员本人 / 具备权限的管理者 | 更新成员名片 | 适合昵称别名、职务等轻量展示信息 |
| `updateParticipantMetaData` | 业务层按权限控制 | 更新业务扩展字段 | 适合部门、等级、购买状态等业务信息 |

### 事件流

| 阶段 | 参与方 | 关键动作 |
|------|--------|----------|
| 权限确认 | 当前用户 | 根据本地角色决定是否展示治理入口和可执行动作 |
| 发起操作 | 房主 / 管理员 / 成员本人 | 发起角色调整、移出成员、资料维护或设备相关协作 |
| 状态同步 | 房间状态 / 业务层 | 角色、成员去留和资料信息同步回成员列表与界面 |
| 结果反馈 | 客户端 | 根据成功、失败、无权限等结果更新提示和按钮状态 |

### 状态与数据

| 数据 / 状态 | 说明 |
|-------------|------|
| `localParticipant.role` | 当前用户的成员角色，用于决定本地是否展示治理入口 |
| `participant.role` | 成员角色，用于判断房主、管理员和普通成员 |
| 成员在会状态 | 用于判断成员是否仍在当前会议中，以及移出成员后的结果同步 |
| `nameCard` | 成员名片，可承载昵称别名、职务等展示信息 |
| `metaData` | 业务扩展信息，可承载部门、等级、购买状态等业务字段 |
| 当前治理动作结果 | 表示成功、失败、无权限、目标已离会等结果 |
| 管理员数量限制 | 应以产品配置或服务端规则为准，业务层不要写死 |

### 角色能力矩阵

| 能力 | 房主 | 管理员 | 普通成员 | 说明 |
|------|------|--------|----------|------|
| 查看成员列表 | 支持 | 支持 | 支持 | 所有在会成员通常都可查看基础成员信息 |
| 设置 / 撤销管理员 | 支持 | 不支持 | 不支持 | 只有房主可以调整管理员体系 |
| 转让房主 | 支持 | 不支持 | 不支持 | 转让后新旧房主权限应立即切换 |
| 移出成员 | 支持 | 支持 | 不支持 | 结果应同步回成员列表和房间状态 |
| 关闭指定成员设备 | 支持 | 支持 | 不支持 | 属于成员级设备治理，详细规则见 `conference/room-moderation` |
| 单独禁言成员 | 支持 | 支持 | 不支持 | 属于成员级消息治理，详细规则见 `conference/room-moderation` |
| 全体禁言 | 支持 | 支持 | 不支持 | 属于房间级会控；通常只约束普通成员 |
| 全体禁用设备 | 支持 | 支持 | 不支持 | 属于房间级会控；通常只约束普通成员 |
| 处理设备开启申请 | 支持 | 支持 | 不支持 | 包括审批或拒绝普通成员的设备开启申请 |
| 邀请成员开启设备 | 支持 | 支持 | 不支持 | 包括邀请成员开启麦克风、摄像头或屏幕分享 |
| 申请开启受限设备 | 不需要 | 不需要 | 支持 | 普通成员在设备受限时可发起申请 |
| 修改任意成员名片 | 支持 | 支持 | 不支持 | 普通成员通常只能维护自己的名片 |
| 修改自己的名片 | 支持 | 支持 | 支持 | 适合昵称别名、职务等轻量展示信息 |

> 这张矩阵从角色视角总结“谁能发起什么动作”。其中 `closeParticipantDevice`、`disableUserMessage`、`disableAllMessages`、`disableAllDevices`、设备申请与设备邀请等能力的详细规则、状态联动和结果处理，仍应归到 `conference/room-moderation`。

### 角色规则与能力边界

1. **通用会议同一时间通常只有一个房主** —— 房主是成员治理的最高权限角色。
2. **只有房主可以设置管理员、撤销管理员和转让房主** —— 管理员不能修改角色体系本身。
3. **管理员数量限制应以产品配置或服务端规则为准** —— 业务层不要在客户端写死上限。
4. **成员重新入会后，应以最新同步到的角色为准** —— 不要继续沿用上一次会议中的临时角色状态。
5. **角色判断应优先使用 `participant.role` 或 `localParticipant.role`** —— 不要依赖 `adminList` 或界面展示文案。
6. **角色视角可以描述设备协作和房间级会控权限，但详细能力仍归对应 slice** —— 在角色常见操作里可以说明谁能关闭成员设备、谁能处理设备申请或发起设备邀请，也可以说明谁能发起全体禁言、全体禁用设备等房间级会控；但这些能力的详细规则、状态联动和结果处理，仍应归到 `conference/room-moderation`。即使入口放在成员列表里，也只是入口复用，不改变能力归属。

### 状态机

```text
idle
  → role-checked
  → applying-action
  → waiting-sync
  → synced
  → idle

applying-action
  → denied
  → failed
  → idle
```

## 最佳实践

### ✅ ALWAYS

1. **同时在界面层和调用层做权限校验** —— 普通成员不可见的治理入口应在界面层隐藏；真正发起接口时仍要处理无权限、目标已离会、申请或邀请超时等失败情况。
2. **统一通过 `participant.role` 或 `localParticipant.role` 判断角色** —— 房主、管理员和普通成员的判断都应来自真实成员状态，而不是 `adminList`、缓存字段或展示文案。
3. **把角色治理、成员资料和房间级会控分层处理** —— 设置管理员、转让房主、移出成员属于成员治理；名片和业务信息属于成员资料；全体禁言、全体禁用设备等属于房间级会控。
4. **让治理结果和设备协作结果都回流到界面** —— 角色变化、成员离会、设备被关闭、申请被处理、邀请被接受或拒绝后，成员列表、入口显隐、待处理列表和结果提示都应同步更新。
5. **让角色规则跟随房间实时状态变化** —— 通用会议同一时间通常只有一个房主；成员离会再入会后，应以最新同步到的角色为准。
6. **让管理员数量限制和业务字段规则来自产品配置或服务端约束** —— 管理员上限不要写死在客户端；`metaData` 的字段含义和展示规则也应提前约定。

### ❌ NEVER

1. **不要只在界面层判断权限而忽略调用失败** —— 即使入口已经做了显隐，接口调用仍可能因为无权限、目标离会或超时而失败。
2. **不要依赖 `adminList`、本地缓存或展示文案判断角色** —— 通用会议不单独维护管理员列表，角色判断应回到成员状态本身。
3. **不要把一次点击或一次接口返回当成最终结果** —— 仍需等待角色变化、成员去留、设备协作结果等真实状态回流。
4. **不要把成员治理和房间级会控写成一套能力** —— 关闭成员设备、全体禁言、全体禁用设备等即使从角色视角可见，详细规则仍应归到 `conference/room-moderation`。
5. **不要长期保留临时角色状态或写死管理员上限** —— 成员重新入会后角色应重新同步，管理员数量限制应以产品配置或服务端规则为准。
6. **不要把 `metaData` 当成无结构字符串自由拼接** —— 业务侧应提前约定字段含义和展示方式，避免成员资料不可控。

## 排障指南

### 常见问题

| 问题 | 表现 | 处理建议 |
|------|------|----------|
| 管理入口展示错误 | 普通成员看到了管理员或房主操作入口 | 检查本地角色判断是否基于 `participant.role` / `localParticipant.role`，而不是缓存或展示文案 |
| 设置或撤销管理员后标签未更新 | 操作成功后，成员列表仍显示旧角色 | 检查角色变化是否已回流到成员列表与入口显隐逻辑 |
| 房主转移后权限混乱 | 新旧房主的可执行操作没有及时切换 | 检查房主变更后的本地角色刷新和按钮显隐是否同步更新 |
| 移出成员后仍显示在列表中 | 成员已被移出会议，但列表里还存在 | 检查成员离会结果是否已同步到 `participant-list` 和 `room-lifecycle` |
| 成员重新入会后角色异常 | 用户重新进入会议后仍沿用上次角色 | 检查是否错误缓存了临时角色，重新入会应以最新同步角色为准 |
| 名片或业务信息更新后展示不一致 | 列表、挂件或资料卡展示不同步 | 检查 `nameCard`、`metaData` 是否使用同一份成员资料来源 |

### 排障流程

```text
发现 参会人管理与角色治理 相关问题
├── 第 1 步：确认问题属于成员角色治理，而不是成员展示或房间级会控
├── 第 2 步：检查当前用户角色和目标成员角色是否已正确同步
├── 第 3 步：确认设置管理员、转让房主、移出成员、资料更新等动作后，结果是否已回流到成员列表和入口显隐
├── 第 4 步：检查是否错误依赖 adminList、缓存角色或本地写死的管理员数量限制
└── 第 5 步：若仍异常，再回查 participant-list / room-moderation / room-invite / room-lifecycle 的衔接是否正确
```
